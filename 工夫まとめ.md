# レッスンレポート自動化ツール — 工夫まとめ

> 課題提出用ドキュメント

---

## 1. 4つのパーツの決め方

このツールは **「Notionに書いたら、あとは全部自動」** という体験を実現するために設計した。  
そのためにデータの流れを一本の川として整理し、4段階に分解した。

```
Notion（記録） → 取得 → 変換 → 配布
```

| パーツ | 役割 | 採用した技術 | 選んだ理由 |
|---|---|---|---|
| パーツ1 — トリガー | いつ動かすかを決める | GitHub Actions（cron + workflow_dispatch） | サーバー不要・月2,000分の無料枠で十分・コードで管理できる |
| パーツ2 — データ取得 | Notionの記録を読む | Notion API（`@notionhq/client`） | 日々のレッスン記録をすでにNotionに書いているため、ここから引っ張るのが最も自然 |
| パーツ3 — 変換 | テキスト → 画像にする | Playwright（HTML→PNG） | CSSでデザインしたレポートをそのままPNG化できる。メール添付・LINE送信いずれにも対応できる汎用性がある |
| パーツ4 — 配信 | 自分のLINEに届ける | LINE Messaging API | 生徒に渡す前にまず自分がチェックする段階のため、LINEへのプッシュ送信が最もシンプルで確認しやすい |

### 選定プロセスで外したもの

- **メール（SMTP）**：最初に検討したが、LINEの方が通知に気づきやすく確認もしやすいと判断し採用しなかった。
- **Zapier / Make などのノーコードツール**：カスタマイズ性が低く、Playwright によるHTML→PNG変換が難しいため不採用。
- **AWS Lambda / Vercel**：コストと設定の複雑さがGitHub Actionsより大きいため不採用。

## 2. デザインの工夫

### 2-1. 受け取る生徒の年齢層に合わせた「読みやすさと温かさ」

生徒は小学生から大人（20〜40代）まで幅広い。  
全員が「一目でわかる」「もらって嬉しい」と感じるデザインを目指した。

- **カラー**：深緑（forest green）をブランドカラーに採用。「成長」「安心感」を連想させる色で、ドラム講師らしい落ち着きを表現した。ベージュ（`#e8e2da`）の背景と合わせることで温かみを加えた。
- **フォント**：見出しに **Playfair Display**（セリフ体）、本文に **Noto Sans JP** を使用。スタイリッシュさと日本語の読みやすさを両立した。
- **サイズ**：横幅390px（スマホ画面幅）に固定。LINEやInstagramストーリーに近いサイズ感で、生徒がスマホで受け取ったときにそのまま見やすい。

### 2-2. 情報の優先順位をレイアウトで表現

```
ヘッダー（誰の・いつのレポートか）
↓
課題曲バンド（今取り組んでいる曲）
↓
本日のレッスン内容（全幅）
↓
よかった点（カード）
↓
改善ポイント（カード）
↓
次回予定日 + 次回レッスン内容（薄緑でハイライト）
↓
フッター（「お疲れさまでした」）
```

よかった点と改善ポイントをそれぞれ独立したカードとして縦に並べることで、ポジティブなフィードバックと改善点が視覚的に区別しやすい構造にした。

### 2-3. ラベルを「日本語主 + 英語小」に統一

当初ラベルは英語のみだった（例：*Today's Lesson*）。  
UIレビューで「一瞬読み取りに時間がかかる」という指摘を受け、  
`本日のレッスン Today's Lesson` のように日本語を主、英語を補助として並べる表記に変更した。

## 3. ぶつかった課題とその解決方法

### 課題① GitHub Actions 上で日本語が文字化けした

**問題**：Playwright でPNGを生成すると、日本語テキストがすべて□（豆腐）になって表示されていた。

**原因**：GitHub ActionsのUbuntuランナーには日本語フォントがデフォルトでインストールされていない。

**解決方法**：ワークフローの中にフォントインストールのステップを追加した。

```yaml
- name: 日本語フォントをインストール
  run: |
    sudo apt-get install -y fonts-noto-cjk
    sudo fc-cache -fv
```

Noto Sans CJK（`fonts-noto-cjk`）をインストールし、フォントキャッシュを更新することで文字化けが完全に解消された。

---

### 課題② Notionのプロパティ名とコードの変数名が一致しておらず、データが空欄になっていた

**問題**：レポートを生成しても、生徒名・レッスン内容・よかった点などがすべて空白で表示された。

**原因**：`fetch-notion.js` でNotion APIのレスポンスから値を取り出す際に使っているプロパティ名（例：`props['今日のよかったところ']`）が、実際のNotionデータベースのカラム名（例：`今日の良かったところ`）と一致していなかった。プロパティ名は完全一致でなければ `undefined` になるため、データが取れずすべて空欄になっていた。

**解決方法**：Notionデータベースを開いて実際のカラム名を一つずつ確認し、コード内のプロパティ名をすべてNotionの表記に合わせて統一した。

```js
// 修正前（コード側の名前がNotionと違っていた）
goodPoints:   props['今日の良かったところ']?.rich_text?.[0]?.plain_text ?? '',
improvements: props['もっと良くできる点']?.rich_text?.[0]?.plain_text ?? '',
nextLesson:   props['次のレッスン内容']?.title?.[0]?.plain_text ?? '',

// 修正後（Notionのカラム名に合わせた）
goodPoints:   props['今日のよかったところ']?.rich_text?.[0]?.plain_text ?? '',
improvements: props['もっとよくできる点']?.rich_text?.[0]?.plain_text ?? '',
nextLesson:   props['次回のレッスン内容']?.title?.[0]?.plain_text ?? '',
```

エラーメッセージは一切出ず、ただ空欄になるだけなので原因の特定に時間がかかった。**Notionのプロパティ名は漢字・ひらがな・送り仮名まで完全一致が必要**という点が最大の落とし穴だった。

---

### 課題③ LINE Messaging API にローカルのPNG画像を直接送れない

**問題**：PNGを生成した後、そのファイルをそのままLINEに送ろうとしたが、LINE APIは**公開されたHTTPS URLしか受け付けない**。ローカルのファイルパスは使えない。

**原因**：LINE Messaging API の画像メッセージは `originalContentUrl` に公開URLが必要。

**解決方法**：生成したPNGをGitHubリポジトリにコミット・プッシュし、`raw.githubusercontent.com` の公開URLを使って送信する方法を採用した。

```
output/2026-04-13-1.png をコミット
↓
https://raw.githubusercontent.com/{user}/{repo}/main/output/2026-04-13-1.png
↓
LINE API に渡す
```

追加の外部サービス（S3、Cloudinaryなど）を使わずに解決できる最もシンプルな方法だった。

---

## まとめ

このツールを作ることで、**毎回手動で書いていた生徒へのフィードバック共有が、Notionに記録するだけで自動化**できるようになった。  
技術的な課題（文字化け・プロパティ名の不一致・画像URL）はすべて実際に動かしてみて初めて気づいたものが多く、エラーと向き合いながら一つずつ解決していくプロセスが、最も多くの学びになった。
